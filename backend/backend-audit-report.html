Backend Audit Report

Scope: backend/ only
Audit date: 2026-04-13
Reviewer focus: architecture, layering, async/await correctness, exception handling, validation, auth/session handling, database safety, security, scalability, and maintainability.

2 Critical findings
12 High findings
10 Medium findings

Executive Summary

The backend is functional but architecturally fragile. Core concerns such as authentication, messaging, storage, and realtime signaling are tightly coupled into top-level route files, and several trust boundaries are not enforced server-side.

The biggest risks are unauthorized data operations, unvalidated socket events, session/token weaknesses, and race conditions around chat and call creation. The project also lacks the structure needed to scale safely: there is no route/controller/service/repository separation, no centralized validation or error policy, and schema/index design does not protect data integrity.

Top Risks

  • Any authenticated user can delete any message by ID.
  • Socket events trust client payloads and allow unauthorized room joins and forged events.
  • Refresh token handling is inconsistent and does not support rotation, revocation, or DB revalidation.
  • Chat and call creation rely on check-then-insert flows without uniqueness guarantees.
  • The codebase is monolithic, making security fixes and regression-safe changes harder than necessary.

Detailed Findings

Critical

1. Message deletion is unauthorized

File: backend/app.js:405-412

Explanation: DELETE /messages/:messageId deletes by primary key only. It does not verify that the caller owns the message or even belongs to the related chat.

Impact: Any authenticated user who learns a message ID can delete another user's data.

Recommended fix: Load the message, resolve the chat, verify sender or chat membership, and consider soft-delete plus audit logging.

Critical

2. Socket events trust client input without authorization checks

File: backend/socket.js:59-145

Explanation: joinRoom, chatMessage, deleteMessage, and call-signaling events accept client-supplied chat IDs, sender IDs, companion IDs, and targets without checking database membership or ownership.

Impact: Clients can join arbitrary rooms, spoof messages, broadcast deletes, and interfere with calls outside their authorization boundary.

Recommended fix: Resolve authorization on the server for every socket event, derive senderId from socket.data.user, and reject room joins or emits when the user is not a verified member.

High

3. The backend is monolithic and lacks route/controller/service/repository separation

Files: backend/app.js:70-537, backend/auth.js:84-278, backend/bucket.js:22-41, backend/call.js:3-15

Explanation: Route handlers directly mix HTTP concerns, auth, database queries, S3 access, and domain logic. There is no consistent controller/service/repository boundary.

Impact: Security fixes, testing, reuse, and performance tuning become error-prone because every endpoint is a custom implementation.

Recommended fix: Split transport, application, and persistence concerns. Introduce routers, controllers, services, repositories, and shared middleware for auth, validation, and error handling.

High

4. Module format is inconsistent and relies on Node's typeless reparsing fallback

Files: backend/package.json:1-37, backend/app.js:9,19, backend/auth.js:10, backend/socket.js:1-12, backend/call.js:1-3, backend/mailer.js:1-11

Explanation: The package has no "type" declaration, but some files use CommonJS and others use ES module syntax. Node 22 reparses these files with a warning and added overhead.

Impact: Startup behavior is environment-dependent, tooling gets confused, and deployment on stricter runtimes can break.

Recommended fix: Standardize on one module system. Either declare "type": "module" and convert fully, or move ESM files to CommonJS.

High

5. Server startup does not fail fast or wait for database readiness

Files: backend/app.js:37-41,541-543, backend/auth.js:45-49,282-284

Explanation: MongoDB is connected in an unguarded async IIFE, while both servers start listening independently. Connection failures are not handled explicitly.

Impact: The app can accept requests before persistence is ready, leading to noisy failures and partial startup states.

Recommended fix: Bootstrap through a single async startup function that awaits DB connection, logs failures clearly, and exits on fatal initialization errors.

High

6. Refresh token and session handling are insecure and inconsistent

File: backend/auth.js:68-77,240-278

Explanation: Refresh tokens are signed for one week but the cookie max age is one hour. Tokens are not rotated, not persisted, not revocable server-side, and refresh trusts old JWT payload data instead of reloading the user from the database.

Impact: Session behavior is unpredictable, revoked or changed users can stay authorized, and stolen refresh tokens remain useful until expiry.

Recommended fix: Store refresh sessions server-side, rotate on use, revoke on logout, align cookie lifetime with token lifetime, and rebuild access-token claims from fresh DB state.

High

7. Login does not enforce email verification

File: backend/auth.js:98-130

Explanation: The login flow validates credentials but never blocks users whose isVerified flag is still false.

Impact: The email verification flow becomes optional in practice, weakening account proof and abuse controls.

Recommended fix: Reject unverified logins or place them into a restricted state until verification succeeds.

High

8. Registration and verification flow is unsafe under failure and uses weak verification-code handling

File: backend/auth.js:174-198,201-237

Explanation: Verification codes are generated with Math.random(), stored in plaintext, and sent after the user is already persisted. If email delivery fails, the database still contains a blocked, unverified account. Expired codes trigger user deletion.

Impact: Registration can dead-end users, verification secrets are weaker than necessary, and expired verification attempts can destroy account records unexpectedly.

Recommended fix: Generate cryptographically strong codes, store only a hash, add retry/resend and attempt limits, and use a transaction or compensating cleanup strategy around user creation plus email dispatch.

High

9. Message creation mixes unvalidated parsing with non-atomic storage operations

File: backend/app.js:70-127

Explanation: The route blindly runs JSON.parse(message) on multipart input and then performs S3 upload plus MongoDB save in one handler without compensation. If one step succeeds and the other fails, state becomes inconsistent.

Impact: Malformed requests can trigger 500s, orphan S3 objects can accumulate, and message state can diverge from stored media.

Recommended fix: Validate payload structure before parsing, wrap the flow in clear error handling, and add compensating delete or a durable outbox/job strategy for media persistence.

High

10. Direct-chat creation has no uniqueness or companion validation

File: backend/app.js:239-250

Explanation: POST /chats inserts a new direct chat immediately. It does not verify that the companion exists, block self-chat creation, or ensure a direct chat between the same two users does not already exist.

Impact: Duplicate conversations, invalid memberships, and inconsistent UX are likely as traffic grows.

Recommended fix: Validate the companion user, normalize member ordering for direct chats, enforce a uniqueness constraint, and return the existing chat when present.

High

11. Call creation uses a race-prone check-then-insert pattern

Files: backend/app.js:458-484, backend/models.js:155-174

Explanation: The code performs separate exists checks before inserting a call room, and there is no unique index on roomId or active-call membership state.

Impact: Concurrent requests can create overlapping or duplicate calls for the same users.

Recommended fix: Enforce uniqueness at the database layer and create rooms through an atomic operation or transaction.

High

12. Call disconnect endpoint never completes the HTTP response

File: backend/app.js:505-511

Explanation: DELETE /calls/:callId awaits callDisconnect and then returns nothing.

Impact: Clients can hang until timeout and may retry, producing duplicate side effects.

Recommended fix: Send a deterministic success or failure response after the disconnect logic finishes.

High

13. Message status patching can crash and mis-handle membership checks

File: backend/app.js:317-338

Explanation: The route reads message.chatId before confirming that message exists. It also uses chat.membershipIds.includes(userId), which is unreliable when comparing ObjectIds to strings.

Impact: Missing records can throw, and valid members may be denied or handled inconsistently.

Recommended fix: Return early when the message is missing and compare normalized strings or use MongoDB membership queries instead of in-memory includes.

High

14. Core collections are missing data-integrity and query indexes

File: backend/models.js:3-181

Explanation: Users do not enforce unique email or login, call rooms do not enforce unique roomId, and common filters such as chatId, membershipIds, and roomId are not indexed.

Impact: Concurrency can create duplicates, and read performance will degrade as collections grow.

Recommended fix: Add unique indexes for identity fields and targeted query indexes for frequent lookups.

High

15. There is no centralized validation or error-handling policy

Files: backend/app.js:70-537, backend/auth.js:84-278

Explanation: Validation is ad hoc and duplicated across handlers, while exceptions are either uncaught, converted to raw JSON objects, or depend on framework defaults.

Impact: API behavior is inconsistent, errors are harder to reason about, and security bugs hide in endpoint-specific validation code.

Recommended fix: Adopt a schema validator such as Zod/Joi, validate params/query/body centrally, and add a single error middleware that maps internal failures to safe HTTP responses.

High

16. JWT payloads contain mutable profile data and use inconsistent role casing

File: backend/auth.js:116-124,223-231,253-259

Explanation: Access and refresh tokens embed description, profilePicture, isVerified, and role values that differ between flows (user vs USER).

Impact: Cookies grow unnecessarily, authorization data becomes stale, and downstream role checks can become inconsistent.

Recommended fix: Keep JWT claims minimal and stable, typically user ID, token ID, and maybe role; derive mutable profile data from the database when needed.

Medium

17. The users list route has a key-type bug and ineffective filtering

File: backend/app.js:141-147,175-195

Explanation: The map stores chat IDs under an ObjectId key but later reads them using string keys, so lookups can return undefined. The subsequent users.filter(...) result is also ignored.

Impact: Chat metadata can be missing or misassigned, and the returned ordering can be wrong.

Recommended fix: Normalize keys with toString() on both insert and lookup, and assign the filtered result back before sorting or returning.

Medium

18. User search does not await async profile-picture enrichment

File: backend/app.js:223-230

Explanation: Promise.all(...) is called but not awaited before the response is sent.

Impact: Clients may receive unresolved or missing signed URLs.

Recommended fix: Await the Promise.all result or map synchronously before responding.

Medium

19. Message pagination validation is weak and raw errors leak to clients

File: backend/app.js:255-315

Explanation: req.query.limit is clamped without explicit numeric parsing, beforeId is not validated, and the catch block returns the raw error object to the caller.

Impact: Invalid input can produce unstable behavior and expose internal failure details.

Recommended fix: Parse and validate pagination inputs explicitly, and return sanitized API errors only.

Medium

20. The /me handler continues after unauthorized branches and assumes the user document exists

File: backend/app.js:513-536

Explanation: The unauthorized response path does not return, and the code immediately dereferences fields on the database user without checking whether the user still exists.

Impact: Requests can fall through into secondary failures or double-response behavior.

Recommended fix: Return immediately after 401, reload the user safely, and handle not-found cases explicitly.

Medium

21. Profile-picture upload handling is brittle and memory-heavy

File: backend/app.js:43-50,433-455

Explanation: Multer uses memoryStorage() with up to 10 MB per file, and the route does not reject requests where req.file is missing after filtering.

Impact: Concurrent uploads can create memory pressure, and invalid files can fail with confusing downstream behavior.

Recommended fix: Validate presence explicitly, prefer streaming or temporary disk storage for larger files, and return a clear 4xx error when MIME validation fails.

Medium

22. CORS and CSRF protections are brittle for a cookie-authenticated API

Files: backend/app.js:21-30,52-64, backend/auth.js:15-24,57-77, backend/socket.js:15-21

Explanation: Allowed origins include raw port values from env vars instead of canonical origins, and state-changing routes rely on cookies without a CSRF token strategy.

Impact: Environment configuration is easy to misapply, and cookie-based auth remains exposed to cross-site request risks.

Recommended fix: Centralize origin configuration as full URLs, validate it at startup, and add CSRF defenses or move to a safer token transport strategy.

Medium

23. Realtime presence and call state are in-process only and break under scale

File: backend/socket.js:13-15,53-58,113-157

Explanation: onlineUsers and rooms are plain in-memory objects. A single user is mapped to one socket ID only.

Impact: Multi-instance deployments lose consistency, restarts drop state, and one tab disconnecting can mark an otherwise active user offline.

Recommended fix: Move ephemeral presence/call coordination to Redis or another shared adapter and support multiple sockets per user.

Medium

24. Schema references are internally inconsistent and one timestamp default is incorrect

File: backend/models.js:68-73,156-169

Explanation: membershipIds are populated with user IDs in application code, but the schema references memberships. Also, callRoomSchema.createdAt uses default: new Date(), which is evaluated once when the schema is created.

Impact: Population semantics are misleading, model intent is unclear, and all call rooms can inherit the same default timestamp.

Recommended fix: Correct the refs to match stored IDs and replace new Date() with Date.now.

Medium

25. Transaction safety is weak across multi-step domain operations

Files: backend/auth.js:181-198,201-213, backend/app.js:100-126,475-483, backend/call.js:3-15

Explanation: Several workflows update multiple systems or documents in sequence without transactions or compensation: registration plus email dispatch, media upload plus message save, and call/message teardown.

Impact: Partial failures leave persistent data in contradictory states.

Recommended fix: Use MongoDB transactions where the database is the source of truth, and apply compensating actions or durable workflows when external systems such as S3 or SMTP are involved.

Low

26. Tooling and hygiene gaps add avoidable maintenance cost

Files: backend/package.json:1-36, backend/auth.js:7,11, backend/app.js:134,282-302

Explanation: There are no test, lint, or build scripts. Some imports and dependencies are unused or redundant, and debug remnants remain in the code.

Impact: Regressions are easier to introduce and harder to catch automatically.

Recommended fix: Add linting and test scripts, remove unused dependencies/imports, and clean debug-only code before release.

Prioritized Action Plan

  • 1. Close the security holes first: add authorization to message deletion, validate every socket event against chat/call membership, and enforce email verification on login.
  • 2. Stabilize auth/session behavior: redesign refresh tokens with rotation and server-side session records, minimize JWT claims, and add CSRF protection for cookie-authenticated writes.
  • 3. Protect data integrity: add unique/index constraints, fix direct-chat and call creation races, and correct schema refs plus timestamp defaults.
  • 4. Introduce platform-level safety rails: centralize validation, add error middleware, and make startup fail fast when configuration or DB connectivity is invalid.
  • 5. Restructure the codebase for maintainability: split routers/controllers/services/repositories, isolate storage and auth utilities, and make multi-step flows transactional or compensating.
  • 6. Improve scalability and operability: move realtime state out of process memory, support multi-socket presence, and add automated tests plus linting.
